验收:e2e_cart_reorder.py——加购后换 token 仍在;商家下架某菜后再来一单跳过它并提示;常买阈值命中正确;下单链路回归绿。

### 59. ETA 动态刷新 + 准时承诺联动

```
在 super-z 仓库深化「预计送达时间」。先读通用约定。

现状:eta_at 在支付时一次性按朴素公式生成,之后不管出餐快慢、骑手远近都不变;用户看到的"预计送达"很快失真,大厂靠动态 ETA 维持信任。
业务规则(已拍板):
- 分段刷新:①商家出餐超承诺时长 → ETA 顺延并推送"商家出餐较慢,预计延后到 X";②骑手接单后按其实时位置到收货点距离重估送达;③到店/取餐节点各刷新一次;
- 与超时赔付(#34)联动:刷新后的 ETA 作为"准时承诺"基准,真实送达晚于承诺触发既有赔付逻辑(不新建赔付,复用);
- 克制:只在明显偏差(>5 分钟)时才刷新推送,避免频繁打扰。
技术要点:services/eta.py 增 recompute(order, rider_pos?);在出餐超时清扫、骑手位置心跳、状态跃迁处调用并写 eta_at;偏差阈值与推送去重;赔付基准取最终 eta_at。
验收:e2e_eta_dynamic.py——出餐超时后 eta 顺延且推送一次;骑手接单后按距离重估;偏差<5分钟不刷;赔付基准用刷新后值。
```

### 63. ⚖️ 反作弊闭环(从"只标记"到"限制 + 复核 + 处置")

```
在 super-z 仓库深化「反作弊」。先读通用约定。⚖️ 处置动作须可申诉、可回滚。

现状:services/risk.py 下单后异步"只标记不拦截"(risk_flags 供后台看);没有对刷单/刷券的实际处置,也没有刷评(同人反复好评/差评、图文雷同)的识别。大厂靠闭环止损,但我们要避免误伤真实用户。
业务规则(已拍板):
- 分级处置(命中风控标记后,阶梯而非一刀切):观察(仅标记)→ 限制(暂停领券/暂停平台补贴,正常下单不拦)→ 冻结(人工复核后再定),每一级都写审计日志且可在后台一键回滚;

- 绝不静默处罚:任何自动限制都给用户可见提示 + 申诉入口;误伤优先放行(宁可漏抓不可错杀,与现有 risk.py 原则一致)。
技术要点:risk_flags 增 level(observe/limit/frozen)+ 处置写 audit/ticket;领券与补贴发放处读该级别做软限制;刷评识别独立评估函数写 review 的 risk 标记;后台风控面板加分级与回滚。
验收:e2e_risk_action.py——命中升级到 limit 后领券被软限但下单正常;回滚后恢复;刷评标记不自动删;每步有审计与申诉入口。
```

## 四、⚖️ 大厂有、但超级赞「故意不做」(价值观护栏,记录在案)

> 列出来是为了防止将来有人"对标大厂"时把这些当缺口补进来。它们与"低抽成·账目透明·不制造焦虑·站在劳动者一边"直接冲突。

- **竞价排名 / 付费买曝光**:搜索与列表排序永远用真实评分×销量×距离,不卖排名。商家花钱买不到靠前。
- **大数据杀熟 / 千人千价**:同一商品同一时刻对所有用户同价,不按设备/画像调价。ETA 与配送费只跟距离/天气,不跟"你愿不愿多付"。
- **二选一 / 独家绑定**:不强制商家独家入驻,不因商家上别的平台而降权。
- **骑手冲单游戏化 / 疲劳诱导**:不做"再跑X单得奖励"的等级冲刺、不用弹窗诱导超时在线。骑手考勤"只统计不考核"(#29),配送费公开透明。
- **好评返现 / 刷单默许**:不诱导好评返现,反作弊(#63)反的就是这个。
- **会员储值 / 预付资质**:见第三辑 #53,永久搁置(预付涉金融资质红线)。
