# 超级赞(chaojizan)

一个群众帮助群众的平台,旨在提高国内群众挣到的钱,向外卖平台等第三方黑心平台宣战。

> **超级赞是「群众帮群众」计划的第一战。** 目标不只是外卖:凡是高抽成、不透明的行业,
> 这套「低抽成 + 账目公开 + 代码开源」的模式都成立。外卖打穿之后,再打下一个。
>
> ⭐ 如果你认同「外卖可以不是这样」,给个 star 就是最简单的支持。
> 也欢迎提 issue / PR / 把它部署到你的城市——或者来帮我们打下一个战场。

**低抽成外卖平台**:商家总负担 5% 封顶(行业平台 20%+),用户端 / 商家端 / 骑手端三端 + 一个后端,
到店团购、透明中心、运营大屏、公开账本一应俱全。

## 平台三原则(产品决策以此为纲)

1. **商家只抽 5%** —— 菜价里没有平台税,点单页明示。**5% 是上限,不是目标:
   哪天 3% 能活,就降到 3%(这行写在这里当立字据)**
2. **配送费 100% 归骑手** —— 按距离计价,平台分文不取,小费也全归骑手,提现零手续费
3. **账目三方透明** —— 每一单"钱去哪了"(商家实收/骑手所得/平台留存)对用户公开可查

透明本身就是与吸血平台竞争的武器。5% 是唯一收入(不卖数据、不竞价排名、不收推广费、
不抽配送费),平台自身收支公开,盈余不分红——优先骑手保障、商家扶持和降费率。

## 宏大路线图

一个行当一个行当地打。每打下一个,就多一套开源的「利润不留在平台手里」的行业模板。

### 第一战 · 外卖(运营中)

点餐、配送、售后、对账全流程已跑通并上线运营:23 个餐饮品类、到店团购(核销才收 2%)、
拼单、邀请有礼、骑手实名与保障、商家阶梯降佣(单量越大费率越低,最低 4%)、
反作弊与食安投诉、每日账务自检、公开账本哈希链。近期战役:

- [ ] 微信支付正式商户号联调(代码就位,模拟支付与真实回调同一幂等入口)
- [ ] iOS 版与用户端 H5 版(微信群传播拉新)
- [ ] 逐城开城:开城清单运营 + 城市骑手/商家招募
- [ ] 分账自动化(微信服务商资质,货款直达商家不经平台沉淀)

### 同场加映 · 到店团购(运营中)

低价引流到店,核销才收 2%;未使用随时全额无理由退,不靠「过期不退」摸钱。

### 第二战 · 打车(筹备中)

司机每单被抽走两三成,高峰还有乘客看不见的差价。我们的版本:司机不被抽走三成车费,
计价规则公开,账目同一套透明框架。

### 以后的仗,一场一场打

凡是抽成不透明、靠信息差吸血的行当,都值得用「低抽成 + 账目公开 + 代码开源」重做一遍:

| 行当 | 行业现状 | 我们要做的样子 |
|---|---|---|
| 跑腿 | 平台抽成 25% 起,小哥风里雨里拿小头 | 跑腿费给跑腿的人,平台只收零头 |
| 家政 | 中介两头收费,阿姨月薪被抽两到四成 | 阿姨的钱不过中介的手 |
| 维修 | 上门费+虚报故障,"小病大修"成行规 | 明码标价,不搞小病大修 |
| 货运 | 司机先交会员费,算法一路压运价 | 不收会员费,不用算法压价 |
| 零工 | 劳务中介层层转包,日结被截一两成 | 日结工资一分不被中介截 |

## 架构

```
chaojizan/
├── apps/
│   ├── user_app/        # 用户端(Flutter):品类逛店、下单、追踪、售后、团购
│   ├── merchant_app/    # 商家端(Flutter):接单出餐、菜品、对账、营销、打印
│   └── rider_app/       # 骑手端(Flutter):抢单、导航、钱包、实名认证
├── packages/shared/     # 三端共享:模型、API 客户端、登录/开屏/推送/地图
├── server/              # 后端(FastAPI + PostgreSQL/PostGIS + Redis)
│   └── app/
│       ├── state_machine.py   # 订单状态机(所有状态变更的唯一入口)
│       ├── services/          # 结算/审计/账本锚点/风控/营销/定价……
│       └── routers/           # 业务路由 + 公开数据(screen/transparency)
├── web/                 # 官网 + 运营大屏 + 透明中心(React + Three.js + ECharts)
├── witness/             # 社区见证节点脚本(任何人可跑,复算公开账本)
└── deploy/              # 生产部署(compose 全栈:api/nginx/frpc,内网穿透自持)
```

**订单状态流**:待支付 → 待接单 → 制作中 → 待取餐 → 配送中 → 已送达 → 已完成
(骑手在「制作中/待取餐」阶段抢单;每次流转写入 order_events 审计表)

```bash

python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
uvicorn app.main:app --reload          # http://localhost:8000/docs 看接口文档

# 另开终端灌演示数据
cd server && source .venv/bin/activate && python -m scripts.seed
```

## 关键设计决策

- **金额一律用「分」存整数**,杜绝浮点误差
- **状态机集中管控**:非法流转直接 409,每次变更写审计表
- **账本只追加不修改**:冲账=追加负数行;每日核账恒等式(e2e 篡改 1 分钱必被抓出)
- **防超卖/防双花/抢单防冲突**:一律数据库条件 UPDATE,并发安全在存储层保证
- **支付幂等**:模拟支付与微信回调走同一入账函数,联调只换通道不换逻辑
- **公开数据无个案**:大屏/透明中心全部平台级聚合,手机号打码、坐标只到城市级
- **e2e 全链路回归**:`make test` 三十余套端到端测试,跑在真实 HTTP 接口上
